9. Thiết lập quy trình làm việc nhóm với Git
Vào tuần thứ 6, sau khi đã lập xong kho lưu trữ (repository) trên GitHub và kéo đủ các thành viên vào, nhóm mới nhận ra một vấn đề khá lớn. Nếu cả ba đứa cùng đẩy thẳng code vào nhánh main, kiểu gì code cũng bị đè lên nhau, lỡ có lỗi thì không biết đường nào mà khôi phục lại. Để tránh "toang" đồ án, cả nhóm ngồi lại và thống nhất áp dụng một quy trình dùng Git đơn giản, phù hợp với nhóm 3 người.
a. Cách chia nhánh (Branching)
Nhóm quyết định dùng 3 nhánh chính, phân chia vai trò rõ ràng:
Nhánh main: Đây là "bản sạch", chỉ chứa code đã chạy thử và ổn định. Không commit và push nếu chưa final vào đây. Chỉ khi nào ráp xong một chức năng hoàn chỉnh mới được gộp (merge) vào.
Nhánh develop: Đây là nhánh nháp chung của cả nhóm. Mọi người làm xong phần của mình thì đẩy code vào đây để ghép nối với nhau. Duy nhất một người trong nhóm sẽ có trách nhiệm đẩy file này sang nhánh main của repo.
Nhánh feature/...: Nhánh làm việc riêng của từng người. Ai code chức năng nào thì tự tạo một nhánh mới từ nhánh develop. Tên nhánh được đặt theo quy tắc: feature/[tên-thành-viên]-[tên-chức-năng].
Bảng phân công các nhánh thực tế của nhóm:
D2. Quy ước viết lời nhắn khi lưu code (Commit Message)
Để sau này mở lịch sử ra đọc còn hiểu được ngày xưa mình đã sửa cái gì, nhóm chốt luôn một form ghi chú chung: <Loại_thao_tác>(<Phần_bị_ảnh_hưởng>): <Mô tả ngắn gọn>
Một số luật ngầm nhóm tự đặt ra với nhau:
Nên lưu code thường xuyên. Cứ code xong một hàm hay logic nhỏ nào chạy được là commit luôn cho chắc ăn.
Nếu code đang báo lỗi (lỗi biên dịch) thì cấm không được gộp vào nhánh develop của nhóm.
Ghi chú phải viết dễ hiểu, người khác đọc lướt qua cũng biết là đang sửa cái gì mà không cần phải mở file ra soi.
D3. Các bước làm việc hàng ngày của nhóm
Cả nhóm tuân theo 7 bước:
Bước 1: Tải code mới nhất của cả nhóm về máy mình bằng lệnh: git pull origin develop
Bước 2: Tạo một nhánh làm việc mới cho mình bằng lệnh: git checkout -b feature/ten-thanh-vien-ten-tinh-nang
Bước 3: Bắt đầu code, chạy thử trên máy. Thấy ổn thì lưu lại bằng hai lệnh: git add <tên_file> rồi git commit -m "feat(scope): lời nhắn"
Bước 4: Đẩy nhánh của mình lên GitHub bằng lệnh: git push origin feature/ten-nhanh
Bước 5: Lên trang GitHub, tạo một yêu cầu gộp code (Pull Request) từ nhánh của mình vào nhánh develop. Nhờ một đứa trong nhóm đọc lướt qua code xem có vấn đề gì ngớ ngẩn không, ok thì comment xác nhận.
Bước 6: Khi cả nhóm đã đồng ý, tiến hành gộp code vào develop. Nếu lúc làm có lỡ tạo nhiều commit lắt nhắt quá thì dùng tính năng "Squash and merge" để gộp gọn lại cho đỡ rác lịch sử.
Bước 7: Gộp xong thì xóa nhánh cũ đi cho gọn bằng lệnh: git branch -d feature/ten-nhanh ở máy mình, trên GitHub thì bấm nút xóa là xong.
D4. Xử lý "sự cố" đụng code (Conflict)
Conflict là cái cảnh hai người cùng sửa chung một dòng trong một file. Tuần vừa rồi nhóm dính chưởng luôn: Việt Anh và Sơn làm sao mà cùng mở file config.h ra thêm code. Việt Anh thêm cái địa chỉ cho màn hình OLED, còn Sơn thêm địa chỉ cho cảm biến nhiệt độ. Lúc gộp code vào nhánh chung thì GitHub báo lỗi không cho gộp.
Cách mấy đứa xử lý:
Dùng lệnh git status để xem file nào đang báo dòng đỏ chữ "(both modified)".
Mở file đó ra, tìm mấy cái dấu ngoặc nhọn <<<<<<<, =======, >>>>>>> mà thằng Git nó tự sinh ra.
Hai đứa gọi điện chốt với nhau là sẽ giữ lại cả hai dòng code này vì nó không liên quan gì đến nhau.
Xóa hết mấy cái dấu ngoặc nhọn thừa thãi đi.
Lưu file lại, gõ lệnh git add config.h rồi git commit báo là đã giải quyết xong xung đột, sau đó đẩy lại lên GitHub.
Bài học nhớ đời sau vụ này: Nhóm quyết định tạo một file cấu hình riêng (như pins_config.h) chứa toàn bộ các chân cắm và địa chỉ.

Tuy nhiên, các mô tả chung chung này không đủ cơ sở kỹ thuật để xây dựng kịch bản kiểm thử (test case) định lượng ở giai đoạn nghiệm thu.
Để đảm bảo tính khách quan và khả năng kiểm tra được (testability) của từng tiêu chí, nhóm đã tiến hành chuẩn hóa danh sách yêu cầu thành một Tài liệu đặc tả yêu cầu hệ thống (SRS) rút gọn Tất cả các yêu cầu được phân loại chặt chẽ thành ba nhóm:
Yêu cầu chức năng (Functional Requirements - FR)
Yêu cầu phi chức năng (Non-Functional Requirements - NF)
Ràng buộc thiết kế và triển khai (Constraints - C)
Mỗi yêu cầu đều bắt buộc phải đi kèm với Điều kiện kiểm tra cụ thể (Pass/Fail criteria) nhằm phục vụ trực tiếp cho công tác viết test case và chạy thử nghiệm cuối kỳ.

FR-03 | Chức năng | Khi nồng độ bụi PM2.5 ở mức trung bình (từ 12 đến 55 µg/m³): Kích hoạt LED vàng bật sáng, đồng thời tắt còi chip (buzzer OFF). | Thử nghiệm thực tế bằng cách tạo môi trường khói nhẹ (đốt nhang cách cảm biến ~50cm) để xác nhận sự chuyển đổi trạng thái LED và buzzer.
FR-04 | Chức năng | Khi nồng độ bụi PM2.5 ở mức an toàn (< 12 µg/m³): Kích hoạt LED xanh bật sáng, đồng thời tắt còi chip (buzzer OFF). | Kiểm tra trong môi trường phòng sạch/hộp lọc khí; hệ thống phải duy trì LED xanh và buzzer im lặng.
FR-05 | Chức năng | Hiển thị đồng thời các thông số: PM1.0, PM2.5, PM10, nhiệt độ, độ ẩm và áp suất khí quyển lên màn hình OLED SSD1306 (độ phân giải 128×64 px), tần suất cập nhật dữ liệu (refresh) là 2 giây/lần. | Quan sát trực tiếp trên màn hình phần cứng: Các thông số cập nhật mượt mà theo thời gian thực, không xảy ra hiện tượng đóng băng (freeze) giao diện hay rác màn hình.
FR-06 | Chức năng | Khởi tạo và ghi tiến trình dữ liệu vào file data_log.csv trên thẻ nhớ SD theo định dạng chuẩn: timestamp,PM1,PM2.5,PM10,Temp,Hum,Press. Tần suất ghi nhận: 10 giây/dòng. | Trích xuất thẻ nhớ, mở file CSV trên máy tính: Xác nhận cấu trúc các cột, trường thời gian (timestamp) tăng đều sai số không quá $\pm$1s, không có dòng trống ngắt đoạn.
FR-07 | Chức năng | Điều khiển quạt hút tự động: Bật nguồn quạt trước chu kỳ đo 3 giây để làm sạch buồng lấy mẫu và tắt ngay sau khi hệ thống hoàn tất đọc dữ liệu. | Đo dạng sóng tín hiệu chân điều khiển GPIO13 bằng Logic Analyzer hoặc Oscilloscope: Đảm bảo mức logic HIGH trong 3s $\rightarrow$ chuyển LOW khi đọc $\rightarrow$ giữ HIGH sau 7s nghỉ.
FR-08 | Chức năng | Xử lý ngoại lệ mất kết nối: Khi mất tín hiệu từ cảm biến PMS7003 (timeout UART > 5 giây), màn hình OLED phải hiển thị cảnh báo lỗi "PMS ERR", hệ thống cô lập tác vụ và không được tự reset (crash). | Thử nghiệm rút dây truyền nhận (TX) của cảm biến PMS7003 trong lúc đang vận hành; hệ thống phải giữ nguyên trạng thái, báo lỗi lên màn hình sau 5 giây và không bị khởi động lại.
NF-01 | Hiệu năng | Hệ thống đảm bảo tính sẵn sàng cao, vận hành liên tục ổn định trong thời gian $\ge$ 8 giờ liên tục mà không tự khởi động lại hoặc làm mất/sai lệch dữ liệu. | Tiến hành kiểm thử độ bền (Stress test) liên tục qua đêm trong 8 giờ; kiểm tra file log CSV phải đảm bảo đủ $\ge$ 2880 dòng dữ liệu quy chuẩn (8h $\times$ 360 dòng/giờ).
NF-02 | Hiệu năng | Thời gian đáp ứng (Latency) từ thời điểm nồng độ bụi PM2.5 thay đổi vượt ngưỡng đến khi LED chuyển màu và còi kêu không vượt quá 500ms. | Đo kiểm thực tế từ thời điểm tác động nguồn bụi vào buồng đo đến khi hệ thống thay đổi trạng thái phần cứng tương ứng bằng thiết bị ghi hình tốc độ cao hoặc đo xung kích hoạt.
NF-03 | Nguồn điện | Tổng dòng điện tiêu thụ của toàn bộ hệ thống ở trạng thái hoạt động cực đại (ESP32 active + PMS7003 chạy quạt + OLED sáng + buzzer/quạt hút bật) phải $\le$ 900mA ở điện áp nguồn 5V. | Mắc nối tiếp ampe kế vạn năng hoặc thiết bị đo dòng chuyên dụng vào đường nguồn tổng 5V để giám sát giá trị đỉnh (peak current) trong suốt các kịch bản hoạt động.
NF-04 | Bộ nhớ | Dung lượng tài nguyên sau khi biên dịch (compile) phải nằm trong giới hạn an toàn: Dung lượng Flash $\le$ 80% (tương đương $\le$ 3.2 MB / 4 MB) và dung lượng SRAM $\le$ 70% (tương đương $\le$ 364 KB / 520 KB). | Kiểm tra và trích xuất số liệu thông báo tài nguyên hệ thống từ trình biên dịch (Compiler Output) sau khi build hoàn tất source code bản Release.
C-01 | Ràng buộc | Tổng chi phí đầu tư mua sắm toàn bộ linh kiện cấu thành phần cứng hệ thống không được vượt quá 2.000.000 VNĐ. | Đối chiếu, tổng hợp dựa trên hóa đơn tài chính hoặc lịch sử giao dịch thực tế từ các nền tảng thương mại điện tử.
C-02 | Ràng buộc | Tất cả các module linh kiện trong danh mục thiết kế phải đảm bảo tính sẵn sàng cung ứng tại thị trường trong nước (Hà Nội hoặc các sàn Shopee/Lazada), thời gian vận chuyển $\le$ 7 ngày. | Minh chứng bằng lịch sử log hành trình đơn hàng thực tế trong giai đoạn chuẩn bị vật tư.
C-03 | Ràng buộc | Mã nguồn dự án phải được quản lý tập trung bằng công cụ Git trên nền tảng GitHub; áp dụng mô hình Git Flow (mỗi tính năng phát triển trên 1 nhánh - branch riêng); tổng số commit có ý nghĩa $\ge$ 20. | Kiểm tra trực tiếp lịch sử đồ thị commit (Commit History) và Network Graph trên Kho lưu trữ (Repository) của nhóm.
